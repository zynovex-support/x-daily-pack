# Source Strategy (Theory → Practice)

本项目的“内容质量上限”由信息源头决定；LLM 评分与写作只能放大信噪差异，无法从根本上把低质量源变成高质量洞察。

## 1) 源头分层

建议把源头拆成 4 层，并在 daily pack 里明确标注来源层级：

1) **Tier A：一手源（最高优先级）**
   - 官方发布（模型/产品/政策/系统卡）
   - 规范与协议 changelog（例如 MCP spec 变更）
   - GitHub releases（SDK/框架/推理引擎）

2) **Tier B：权威二手源（精选）**
   - 少数高质量分析周刊/博客（能提供“可复现证据/对比/取舍”）

3) **Tier C：社区雷达（早期信号）**
   - HN、Reddit、X 研究者账号、GitHub trending
   - 需要严格过滤规则，否则噪声会淹没主线

4) **Tier D：新闻聚合（仅补位）**
   - Google News 等
   - 容易混入重复、标题党、财经噪声；应限制配额或只用于“发现线索”

## 2) “不要迷信 RSS”的具体含义

RSS/Atom 只是“传输方式”，不等于高质量内容。

- **适合 RSS 的**：
  - 官方博客有稳定 RSS 且内容结构化
  - Substack/周刊
- **更适合 GitHub releases 的**：
  - 工具、框架、协议 SDK、参考实现（release notes 本身就是高信号）
- **更适合 changelog/页面监控的**：
  - 没有 RSS 但变更重要（协议规范、榜单、定价页等）
  - 注意：部分站点有反爬/挑战（403），自动抓取不稳定，需要替代方案

## 3) 质量审计（可执行原则）

对每个候选源，至少做一次“最近 30 条抽样”的人工或脚本审计：

- 可用性：HTTP 状态码、Content-Type、是否稳定（403/限流）
- 信噪比：High-signal/Medium/Noise 比例（用标题 + 链接域名 + 是否含实质信息判断）
- 重复度：同一主题/同一链接是否频繁出现
- 覆盖面：是否覆盖你关心的“理论→落地应用→协议/工具→选型对比→监管”

> 只要一个源长期 Noise 偏高，就应被降级到 Radar 或剔除。

## 4) 当前策略在系统里的落点

- **RSS**：见 `config/rss-feeds.json`
- **X 搜索**：见 `config/x-queries.json`
- **配额/筛选/评分**：
  - X/RSS 配比、按来源 cap、X 互动预排序 → `scripts/llm-rank-node.js`
- **写作约束**：
  - 不引用 X 链接、敏感词 blocklist、280 字截断 → `scripts/tweet-gen-node.js`

## 5) 下一步：用“源头审计表”替代主观争论

项目内存在多份外部调研输出（Perplexity 报告），但其中可能包含：
- 把“无 RSS”误判为“站点关停”
- 把“网页 403”误判为“RSS 停用”
- 用“估算”替代真实抽样

因此后续源头决策建议以“可复现审计表”为准，而不是以单次 LLM 报告为准。
